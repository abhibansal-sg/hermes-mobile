# REVIEW-R1-FINDINGS — Hermes Mobile (iOS) + Gateway Bug-Hunt Ledger

**Round:** R1 (5 rounds, exit reason: round cap)
**Confirmed findings:** 97
**Severity split:** P0 = 1 · P1 = 26 · P2 = 47 · P3 = 23
**Scope:** `apps/ios/HermesMobile`, `apps/ios/HermesWidgets`, `tui_gateway/`, `hermes_cli/` (gateway/web/push/device-token/audit/fs layers)

This ledger feeds the **R1 fix loop** and the upstream PR-series quality evidence. Each finding carries a verdict consensus across 3 adversarial judges (REAL / severity-adjusted / refuted). Findings where one judge dissented are flagged in the Consensus column and called out in the FIX-LOOP PLAN for adversarial re-verification.

Severity legend:
- **P0** — broken core feature on the default/production path (ships dead).
- **P1** — data loss, security takeover, permanent UI wedge, or core feature unreachable.
- **P2** — correctness/security gap reachable under realistic conditions; partial loss/leak.
- **P3** — cosmetic, presentation, narrow-edge, or contract-drift-without-current-impact.

# FIX-LOOP PLAN

The 97 findings cluster into a small number of root-cause families. Fixing by family (not by finding) is the efficient path: many P1/P2 entries collapse to one fix. Batches are ordered by leverage (findings retired per change) and risk.

## How to read this
- **Batch leverage** = how many ledger entries the batch retires or de-risks.
- **Re-verify** = the batch interacts with concurrency/ordering/state machines; after the fix, re-run the adversarial judges against the cited findings (not just the unit tests, which in several cases enshrine the wrong contract — see #1, #27).
- **Won't-fix candidates** carry an explicit reason; they still need a one-line decision in the PR.

### BATCH A — Wire-contract drift (gateway ⇄ iOS protocol) — **HIGHEST LEVERAGE**
Retires/affects: **#1 (P0), #4, #13, #27, #47, #58, #59, #46, #76, #81, #82, #84, #86, #87, #33, #70**

Single root cause: the iOS decoders and the gateway emitters disagree on keys, nesting, RPC names, and which event types exist. These are independent one-line-to-small fixes but must land together as one "protocol parity" PR with a regenerated contract fixture, because the existing iOS unit tests bake in the *fictional* contracts (`tool_call_id`/`duration_ms` for #1, nested `usage` for #27) and are green against the wrong wire shape.

Fix actions:
- **#1 (P0):** decode `tool_id` (PRIMARY) with `tool_call_id`/`id` fallback; read `duration_s`. Rewrite GatewayEventTests fixtures to the real wire shape.
- **#4:** add an `error` GatewayEventType case → fail the streaming message + clear `isStreaming` (mirror desktop `failAssistantMessage`). This also unblocks #20 and #26 (error is the only terminal frame on the crash path).
- **#13:** thread `request_id` through `ClarifyRequestPayload`/`PendingClarification` and both `respondClarification` call sites.
- **#27:** call `session.usage` and decode its FLAT result as `UsageStats`; delete the `SessionStatusResult.usage` path. Retire the fictional ContextMeterTests fixture.
- **#47, #81, #82:** add a `session.info` decoder case (update cwd/branch) and decide whether to consume `status.update` (#81) vs keep dropping (won't-fix-with-reason if intentionally minimal).
- **#58:** map `failed`/`interrupted` → `.error` in `SubagentNode.Status`.
- **#59:** read the structured top-level `todos` array from `ToolCompletePayload` instead of re-parsing the truncated preview.
- **#46, #87:** apply `message.complete` `status` (interrupted/error badge) and `reasoning` (when streamed thinking is empty).
- **#76:** surface `command`/`approval_title` from the approval payload.
- **#84:** normalize WS `approve→once` server-side to match REST audit.
- **#86:** decode `resumed` as a `storedSessionId` fallback.
- **#33, #70:** parse the top-level `broadcast_gap` frame field and trigger a REST backfill on a non-zero gap.

**Re-verify:** YES — #1, #4, #13, #27 change live streaming behavior; re-run judges + a real end-to-end turn (not unit tests alone).

### BATCH B — ChatStore streaming/ownership state machine + backfill races
Retires/affects: **#9, #12, #21, #42, #51, #52, #61, #28, #43, #79**

Root cause: `isStreaming`/`backfill()`/`seed()`/`cancelStreaming()` lack post-await re-checks (openToken / `activeStoredId` / foreign-vs-local ownership) and don't clear pending-prompt state. Several findings are the same missing guard reached by different triggers (server restart, socket drop, session switch, foreground).

Fix actions:
- Add a streaming watchdog/teardown for an interrupted LOCAL turn on connection drop (#9, #42) — analogous to the existing `teardownForeignStream()`.
- Re-check `!isStreaming` AND `activeStoredId == storedId` AFTER every REST await in `backfill()` (#12, #21) and thread the open token into `seedTranscript` so both fast-path and chain-tip seeds bail when superseded (#28, #43).
- Guard `seed()`/`cancelStreaming()` against an in-flight adopted foreign mirror (#61) and against wiping pending prompts on reconnect (#51, #52) — expire pendingApproval/Clarification/SecurePrompt on `message.complete` (mirror InboxStore) and on backfill-after-restart.
- Fix the stuck loading spinner after foreign teardown + failed backfill (#79) by surfacing `lastBackfillError` to ChatView instead of an infinite spinner.

**Re-verify:** YES — these are the highest-density concurrency findings; each fix needs the adversarial judges re-run because a naive guard can reintroduce a sibling race (the ledger already contains 3 judge-dissent flags here: #61, #79, #10's neighborhood).

### BATCH C — Composer/QueueStore foreign-mirror + session-scoping + offline-outbox
Retires/affects: **#2, #17, #18, #10, #50, #29, #30**

Root cause: the composer derives its entire mode from `chatStore.isStreaming` with no access to local-vs-foreign ownership or session identity, and QueueStore prompts are global + dequeued-before-success.

Fix actions:
- Give the composer access to `localTurnInFlight`/`streamingIsForeign` so QUEUE/STOP mode and the user-action guards (#30) reflect *local* ownership, and route STOP to the streaming runtime (#2).
- Stamp `QueuedPrompt` with session identity; drain only into the matching session (#17).
- Re-enqueue (don't dequeue) on a send that neither streamed nor was genuinely accepted; gate drain on agent idleness (#10, #50). Add a drain trigger for foreign-complete (#29).
- Make the offline outbox reachable: route a disabled-because-offline send to `enqueue`, or show the queue affordance when `!isConnected` with text present (#18).

**Re-verify:** YES — #2/#17 are safety-relevant (misrouted destructive prompts); judges flagged exploitability nuance on #10 and #50.

### BATCH D — Live Activity lifecycle (orphan / reuse / ordering / double-end)
Retires/affects: **#3, #26, #49, #53, #73, #75, #32/#97, #44, #83**

Root cause: LA delivery is N independent unstructured Tasks (no ordering), `end()` doesn't reset `activity`/`registeredLAToken`/`sessionId`, there's no lifecycle owner that ends the activity when the turn it mirrors is discarded, and the server fires two end frames on interrupt.

Fix actions:
- Serialize LA delivery (single serializing actor / AsyncStream drained in order) and cancel in-flight update Tasks in `end()` (#3).
- Add LA-end hooks on disconnect/reconnect/scene-phase and in `reset()`/`cancelStreaming()` (#26, #73); set a `staleDate` so ActivityKit self-stales an orphan.
- On the reuse branch, re-arm `observePushToken`/`registerLAToken` and update `sessionId`; have `end()` reset `registeredLAToken=nil` so re-register isn't suppressed (#49, #53).
- Thread the cancellation token through the LA registration round-trip (#32, #97).
- Server: unregister the LA token in `_finalize_session`/`session.close` and GC by age (#44); collapse the interrupt double-end (#75); fix the elapsed-stamp ordering (#83).

**Re-verify:** PARTIAL — #75, #32, #97 carry judge dissent on whether the bad ActivityKit state is actually reachable; re-verify those three specifically. The rest are deterministic.

### BATCH E — Push/APNs offload + per-event I/O + payload hygiene (server)
Retires/affects: **#8, #64, #22, #65, #93, #85, #88, #82**

Root cause: `_emit→_push_hook` does blocking network + filesystem work synchronously on the agent emit/streaming thread, and the throttle/payload paths leak secrets, drop the approval frame, and over-prune.

Fix actions:
- Offload push/LA sends to a daemon thread or queue (#8); move the per-event `stat()`/registry-read off the hot path and behind the throttle / a cached arm check (#64).
- Bypass the throttle when `needs_approval` is True / on the approval transition (#22).
