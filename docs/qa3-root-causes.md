## S4 — Answer renders ABOVE its prompt; split/duplicated within one view (P0) + S6 — prompt vanished (P0)

**Merge machinery:** `applyRelayItems` (`ChatStore.swift:4976-5200`) rebuilds the WHOLE session from the session-accumulating `RelayItemStore.items` (relay snapshot = ALL accumulated items, `relay/hermes_relay/session_state.py:83-98` `ordered_items()`/`snapshot()`, keyed by `ord` + first-seen tiebreak) and APPENDS it under the preserved untagged history: `preserved = messages.filter { !$0.relayProjected && !consumedEchoIDs.contains($0.id) }; messages = preserved + rebuilt` (5195-5198).

**Root cause (S4):** `consumedEchoIDs` is populated ONLY by `adoptRelayEcho` (4916-4936) — USER rows (matched by `client_message_id`, then text). **Untagged ASSISTANT rows are never consumed.** When a reconnect/resync delivers a snapshot covering turns that the GRDB cache/network seed already painted untagged (this round: WS re-opens at 10:46:31 + 10:47:18 per relay.log — exactly the IMG_2579-2582 capture minute), the rebuilt tail contains `[U1,A1' … Un,An']` (correct order, adopted user ids, `relayAssistantID(forUserRowID:)` 5050-5066) while preserved STILL holds every cache-painted assistant row `[A1 … An]` (users consumed out). Result: each answer exists twice — an ORPHAN cache copy sitting ABOVE the rebuilt user rows, and the correct-order copy at the tail. That is precisely IMG_2579 (orphan answer tail + action row, then the rebuilt user bubble below it, no answer under it) vs IMG_2580 (the correct-order rebuilt copy: user → "Worked ›" → answer) in the SAME session at the SAME minute; and IMG_2582 (scrolled into a copy whose prompt is above the frame → "prompt missing at the bottom"). "Session switch away+back fixed it" — switch resets the item store (`RelaySessionCoordinator.resetItemStoreForSessionSwitch` 612-629) and the cache paint alone is in correct order until the next snapshot re-projects.

**Root cause (S6, prompt durability):** the optimistic user echo exists only in the in-memory `messages` (not durably persisted as settled history). Two loss paths:
1. Failed/ambiguous relay submit → catch path `removeLocalEcho` + `abandonRelayTurnAffordance` (2719-2720) deletes the echo; it is replaced by the outbox row's echo (`presentOutboxEcho` 2735) only if the outbox enqueue succeeds — and the outbox drain (`submitOutboxPrompt`, 2895+) re-presents only when the destination session is the driven one (`outboxRuntimeID(forStored:)` `RelaySessionCoordinator.swift:560-562` HOLDS otherwise). IMG_2584 (Aheli, 10:49): reply echoed, composer IDLE, no affordance — the send left the immediate path (socket not open at that instant, or 4009-busy fallback) into the outbox, and no working surface was raised at send time (the outbox path sets `setStreaming(true)` only at drain, 2937, and appends NO optimistic assistant bubble at all).
2. Session switch mid-turn: `messages` is reseeded for the new session; the echo survives only if the relay later delivers the synthesized `userMessage` item (`_emit_user_message_item`, `relay/hermes_relay/downstream.py:565-600`). IMG_2585 (11:25, CLIProxyAPI): reconnect at 11:25:28 inside a 5s-young turn — the prompt is gone from view while the "Working… · ToolCall 5s" line renders: the agent items re-projected but the user row had not re-landed (its adoption target was deleted by path 1, or the item had not replayed yet). Same family explains IMG_2591 ("??" sandwiched between two working lines in Aheli): turn 1's prompt + turn 2's "??" prompt with neither turn settling (see S8).

**Fix outline (A2):** (a) `applyRelayItems` must consume untagged ASSISTANT rows the projection covers — correlate by turn (user row ordinal/`userOrdinals`, or by matching the assistant row's text prefix against the rebuilt segment) and add them to the consumed set; or (b) invert the merge: when the item store carries a turn, the rebuilt copy is authoritative and the untagged twin is dropped. Add a chronology invariant test in `tests/render_conformance`: replay this round's recorded sequence (submit → items → snapshot-after-reconnect) and assert single-copy strict chronological order, never answer-before-prompt. (c) Echo durability: persist the echo + its `clientMessageID` to the transcript cache at send (write-through) so switch+back paints it even before the relay item lands; and the outbox-mode send must raise the SAME optimistic affordance at enqueue time (bubble + merged working line), not only at drain.

**Refuted:** "RelayItemStore sort instability" — the store sort is stable (`RelayItemStore.swift:38-48`, ord then arrival) and the relay allocates the userMessage ord at SUBMIT before agent items (`downstream.py:593` `ord=state.allocate_ord()`), so wire order is correct; the inversion is created by the iOS merge, not the wire. "SwiftUI duplicate-id glitch" — ids are distinct (adopted echo id vs relay-deterministic ids); the duplication is in the data (`messages` array), verifiable in a render_conformance replay without SwiftUI.

## S7 — Pure blank scrollback (still)

**Mechanism:** the transcript is a plain EAGER VStack rendering only the windowed tail `allRows[windowStart...]` with `windowStart = max(0, allRows.count - windowSize)` (`Views/Chat/ChatView.swift:1062-1070`), bottom-anchored (`defaultScrollAnchor(.bottom)`, clearance spacer 1216-1225). There is NO lazy height estimation — "blank" can only mean the rendered content is SHORTER than the viewport. IMG_2589/2590 show exactly that: a "Worked ›" row + one summarization fragment pinned at the bottom, void above — the transcript was truncated to a few rows mid-session.

**Root cause (the reseed holes that survive R15):** R15 made network/backfill reseeds `.union` (`ChatStore.swift:4666`, `SessionStore.swift:5625/5732/5796/5819`), but: (1) the FIRST-FRAME cache paint still seeds with the DEFAULT policy `.replace` — `chat.seed(normalized:)` at `SessionStore.swift:5541` (`seed(normalized:policy:.replace)` is the signature default, `ChatStore.swift:3707`), and the cache itself is written from PARTIAL windows (backfill persists the fetched tail, `ChatStore.swift:4670-4676`; network seeds save `stored` windows) — so a cache-first re-paint of the SAME session after a reconnect/foreground replaces the full in-memory transcript with the cached tail window; (2) cache-miss still `chat.reset()` then skeleton (5550). Post-truncation, the eager bottom-anchored VStack renders the handful of surviving rows at the bottom → pure void on scroll-up (and the "Load earlier messages" chip only appears when `windowStart > 0`, 1091-1097 — useless when the STORE itself lost the rows). The 10:46-47 reconnect flap + mid-turn auto-compaction (`isActiveSessionCompacting`, ChatView 1160) are the in-round triggers.

**Fix outline (A3):** make same-session cache paints `.union` too (the window is known-partial by construction); never persist a partial window to the cache as if it were the transcript (mark windowed caches and union them on paint); windowing test in render_conformance: tall history → scroll-up never yields a viewport with zero realized rows (content or skeleton, never void — QA-1 B4 extended).

## S8 — Eternal double-working, nothing ever arrives (P0)

**State machine:** turn liveness = `relayTurnLive` (set at send 2645; cleared only by `turnSettled` on `turn.completed` frame, 4985, or wholesale reset 5255-5261) OR any rebuilt row `isStreaming` (`nowStreaming = relayTurnLive || rebuilt.contains { $0.isStreaming }`, 5133). Per-row `isStreaming` derives from item terminality (`segmentStreaming`, 5036) — an item stuck `.inProgress` in the item store keeps its rebuilt row streaming FOREVER.

**Existing fallback:** the QA-2 R12 local-turn watchdog (`armLocalTurnWatchdog`/`fireLocalTurnWatchdog`, `ChatStore.swift:5328-5371`): after `localTurnStaleTimeout` with no item batch, force-settles the STORE flag. Two fatal gaps: (1) it is RE-ARMED by ANY item batch while `isStreaming` (`applyRelayItems` 4990 `if isStreaming { armLocalTurnWatchdog() }`) — IMG_2591's second turn ("??") keeps emitting ToolCall frames, re-arming the watchdog on every pass, so it never fires for turn 1's stuck items; (2) even when it fires it only clears the store flag — the next projection re-derives `nowStreaming == true` from the still-`.inProgress` items and re-lights streaming. There is NO resync-from-relay liveness fallback anywhere (no "no completion within N → fetch snapshot / reconcile" path; the coordinator's resync is connection-drop-driven, `RelaySessionCoordinator.swift:379-406`, not silence-driven).

**Root cause (S8):** dead turns (gateway turn died / relay lost the terminal frame / the submit never drove) leave `.inProgress` items that the projection renders as live working lines indefinitely; the only watchdog is per-store, re-armable by unrelated frames, and does not heal the item store. Relay.log shows the Aheli session (20260720_171346_ad9133) fetched at 10:49:19-29 and 11:28:01 with the turn visibly never completing on device — consistent with a lost terminal frame or a turn the gateway never ran after the flap.

**Fix outline (A4):** per-TURN liveness: track `lastFrameAt` per turn (not per store); no frames for THIS turn + no completion within N → SILENT resync (coordinator `resync{last_seq}` / snapshot re-fetch for the driven session), reconcile the item store against authoritative state; if the authority says the turn is gone → settle the rows (mark terminal with a muted "interrupted" fold, never an error banner — C3). State-machine test: simulate dead turn (frames stop, no completion) + a LIVE second turn streaming — assert turn 1 self-heals within N and the second turn is unaffected (kills the re-arm-by-other-turn bug). Eternal-working must be unreachable by construction.

## S9 — Drawer stuck/snap-back, 3rd recurrence

**Current machinery:** row tap → `sessions.open(summary) { onNavigate() }` (`Views/Drawer/DrawerView.swift:1934`, search 1581, project detail 2626) → the close is a PENDING INTENT: `pendingDrawerReveal` (`Stores/SessionStore.swift:3582-3593`), fired EXACTLY ONCE on first paint OR a 300ms liveness deadline (`open()` 3728-3776; the deadline task is deliberately NOT gated on `openToken`, 3770-3776 — that was the QA-1 B3 fix). Re-opens: `.hermesOpenSessionsIntent` → `drawerState.open()` (`Views/Shell/RootView.swift:85-87`), posted by `PendingIntentRouter.apply/.drainDurable` (`Intents/PendingIntentRouter.swift:65,95` — drained on foreground), plus the drag-settle raw flip (`RootView.swift:1807 setOpenRaw(commitOpen)`).

**Root cause (structural, 3rd recurrence):** dismissal is a one-shot latch consumed at first-paint/300ms, but RE-OPEN edges are unserialized against it — last writer wins. The in-round suspect: a durable `.openSessions` App-Intent job (or legacy import via `importLegacyWork`, PendingIntentRouter:53-55) re-drained on the foreground transition that accompanies "tap during in-flight load" (background → foreground to check the stuck turn) posts `hermesOpenSessionsIntent` AFTER the tap-close → drawer re-opens = the reported snap-back/stuck. QA-2 R2 (40b7e6208) only covered search-result taps; the intent-flush vs intent-close race is untouched. IMG_2592 shows the drawer open over the S8-stuck session (single frame; consistent, not conclusive).

**Fix outline (A5):** serialize drawer intent through a monotonic epoch: the tap-close stamps an epoch; any open-intent (notification route, App Intent drain, reveal-on-paint) older than the current close epoch is dropped until the next user gesture; `drainDurable` completes stale `.openSessions` jobs without re-posting when a newer close intent exists. UI test reproducing tap-during-in-flight-load (incl. a foreground transition mid-load) must show the drawer always dismisses into the session.

## S10 — Projects detail shows SESSIONS 0 (P0)

**Chain:** `ProjectDetailView` reads `projectsStore.sessions(for:)` (`Views/Drawer/DrawerView.swift:2583`, `Stores/SessionStore.swift:6390-6392`) populated by `refreshSessions(for:)` (`SessionStore.swift:6300-6345`): primary `rest.projectSessions(projectId:)` = `GET {mobileAPIPrefix}/project-sessions?project_id=<root>` (`Networking/Rest/RestClient.swift:278-290`, plugin route served at `plugins/hermes-mobile/dashboard/api.py:2800-2855`); on transient failure (401/403/404/5xx — `SessionStore.swift:6268-6277`) fallback `rest.sessionsWithTotal(cwdPrefix: root)` = `GET /api/sessions?cwd_prefix=…` (RestClient.swift:204-256). The screenshot is the EMPTY state (not the error row) → a fetch SUCCEEDED with `[]`.

**Root cause:** the fallback is structurally incapable of listing worktree projects — RestClient.swift's own doc (258-270): "a project's `root` is the folded git-common repo root, while its sessions run in WORKTREES whose cwds are siblings of that root — the `cwd_prefix` LIKE never matches, so the detail list comes back empty even though the count is > 0." hermes-mobile is exactly such a project (dozens of worktrees, incl. under `/Volumes/MainData/Developer/hermes-tmp/worktrees/`). So whenever the primary plugin route does not deliver (path-style resolution to `.legacy` → `/api/project-sessions` 404s — `ServerCapabilities.resolvedPathStyle` defaults `.legacy` until the plugin-mount probe concludes `.available`, `ServerCapabilities.swift:111-112`, and the probe runs on the gateway-connect/settle flows `ConnectionStore.swift:1662/3113` which relay-only operation can skip; OR the plugin route's `_build_project_tree(include_discovered=False)` misses the root and `_project_sessions_server_fallback` returns its silent `empty` when `SessionDB`/tree throws, `api.py:2739-2797`), the detail degrades to a FALSE "No sessions yet" with no error surface.

**Fix outline (A6):** (a) server: `_project_sessions_server_fallback` must fold worktree cwds (resolve each session cwd through `_git_common_repo_root_for_cwd`, `tui_gateway/server.py:1809/1851`, instead of raw `cwd_prefix` LIKE); (b) iOS: never accept fallback `[]` as authoritative when the overview's `sessionCount > 0` — retry the alternate path style (the self-healing 404-retry pattern already exists for push/LA registration, extend it here) and surface a retry state rather than a lying empty state; (c) relay mode: run the capabilities probe regardless of WS transport. Verify on isolated gateway 9130+ (A6) — live gateway 9119 is off-limits for testing.

**Refuted:** QA-2 commit 1676e8bb5 (RestClient path-escape change) — for these two routes the path carries no escapes and the query is preserved via `percentEncodedQuery` identically before/after (verified by diff); behavior-identical. QA-1 touched ServerCapabilities only by a DEBUG test seam (diff 1fcffe3d5^..1fcffe3d5).

## S12 — Notification tap opens Inbox, not the session

**Chain:** tap → `NotificationLaunchCoordinator.didReceive` (`Support/NotificationLaunchCoordinator.swift:194-224`) → `NotificationService.decodeTap` (`Support/NotificationService.swift:615-651`, extracts `session_id` → `.turnComplete(sessionId:)`) → `HermesURLRouter.routePushTap` (461) → `openForPush` (`App/HermesURLRouter.swift:522-555`): `storedId = inbox.storedSessionId(forRuntime: runtimeSessionId) ?? runtimeSessionId` (`Stores/InboxStore.swift:137`); lookup in `sessions.sessions`; on miss: wait-ready → `sessions.refresh()` → lookup; on second miss → `inbox.requestPresentation()` — **the Inbox the owner sees**.

**Root cause:** push payloads carry the RUNTIME session id (doc at HermesURLRouter.swift:444-460), and the runtime→stored translation relies on the INBOX having seen that session (attention items carry both ids). For a `turn_complete` of an ordinary (non-attention) turn the inbox never held the mapping, and any compressed/old session has runtime ≠ stored → both lookups miss → Inbox fallback. The relay knows the stored id (it drives sessions by stored id — `RelaySessionCoordinator.activeStoredSessionID`; durable-state owns origin ids) but the notifier payload does not carry it.

**Fix outline (A8):** relay: add `stored_session_id` to the push `hermes` block (notifier.py compose path) — the relay resolves it at drive time. iOS: `openForPush` resolves via stored id first when present; keep the runtime→stored inbox map as fallback; replace the silent Inbox dump with a resolving state (refresh+retry once) — Inbox only on genuine not-found. Routing unit test (payload with stored id → opens that session) + manual device proof.
